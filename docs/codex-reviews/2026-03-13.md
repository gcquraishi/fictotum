## CRITICAL

### CRITICAL-1 — Public endpoint can trigger paid LLM calls (cost/DoS risk)

- **Where**: `web-app/app/api/figure/[id]/discover/route.ts:12`
- **What**: `GET /api/figure/[id]/discover` performs Anthropic requests when `ANTHROPIC_API_KEY` is present, but has **no auth**, **no rate limiting**, and **no caching**. `id` is fully user-controlled. (`process.env.ANTHROPIC_API_KEY` gate only stops calls when missing, not when deployed.) `web-app/app/api/figure/[id]/discover/route.ts:34`, `web-app/app/api/figure/[id]/discover/route.ts:78`
- **Why it matters**: any anonymous actor can create unbounded spend and/or exhaust function concurrency, degrading the entire site.
- **Suggested fix**:
  - Require auth (at least) or restrict to admin/allowlist.
  - Add rate limiting + request-level timeouts + caching (e.g., cache narrations by `{figureId, candidateHash}`).
  - Cap input sizes (e.g., `id` length) and add a hard “no-network” fallback path that returns 200 with `narrated: false`.

### CRITICAL-2 — “Admin” image regeneration route is not admin-only + allows arbitrary DB mutation

- **Where**: `web-app/app/api/admin/regenerate-image/route.ts:115`
- **What**: `POST /api/admin/regenerate-image` requires only “any authenticated session” (no role/allowlist). Any signed-in user can:
  - trigger expensive Gemini image generation, `web-app/app/api/admin/regenerate-image/route.ts:186`
  - upload to R2, `web-app/app/api/admin/regenerate-image/route.ts:223`
  - update `image_url` for *any* figure/work by ID, `web-app/app/api/admin/regenerate-image/route.ts:252`
- **Why it matters**: privilege escalation (regular users can mutate canonical content) + cost/abuse risk.
- **Suggested fix**:
  - Enforce a real admin gate (e.g., `session.user.email` in an allowlist env var) before generating/uploading/updating.
  - Validate `entityId` format per entityType (canonical IDs / Wikidata QIDs / media IDs) and reject unexpected sizes/characters.
  - Validate all R2 env vars before `.trim()`; otherwise this can throw and return noisy 500s (`R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET_NAME`).
  - Consider a background job queue to avoid long-running requests and repeated retries on client refresh.

## HIGH

### HIGH-1 — Public collection API leaks owner email (PII) and enables enumeration

- **Where**:
  - Public route explicitly states it is public: `web-app/app/api/collections/[id]/route.ts:10`
  - Owner object includes `email`: `web-app/lib/collections.ts:25`
  - Public GET returns full `getCollectionById()` payload: `web-app/app/api/collections/[id]/route.ts:11`
- **What**: any visitor can query `/api/collections/{collection_id}` and receive `owner.email` (and name/image). Collection IDs are time-based + short random suffix, which makes targeted guessing/bruteforce more feasible. `web-app/app/api/collections/route.ts:65`
- **Why it matters**: PII leakage and potential user targeting/spam.
- **Suggested fix**:
  - Do not return email in public responses; return a stable public `owner_handle` or omit owner identity entirely unless explicitly opted-in.
  - Consider a separate “share token” (unguessable UUID) for public access, instead of exposing `collection_id`.

### HIGH-2 — Public collection page uses owner email for ownership checks (forces server data to include PII)

- **Where**: `web-app/app/collection/[id]/page.tsx:68`
- **What**: ownership check (`session.user.email === owner.email`) requires fetching `owner.email` for every viewer, which pushes PII through the rendering path for public pages.
- **Why it matters**: even if the API is fixed later, this page will keep incentivizing the data model to include email in public payloads.
- **Suggested fix**:
  - Store a non-PII owner identifier on the collection (`owner_user_id`), or do an ownership check via an authenticated-only API call.
  - Keep public payloads PII-free; compute `isOwner` server-side without returning `owner.email`.

## MEDIUM

### MEDIUM-1 — Analytics stats endpoint likely needs admin-only authorization (currently “any user”)

- **Where**: `web-app/app/api/analytics/stats/route.ts:38`
- **What**: route checks only `session?.user?.email` (any signed-in user). No admin/role check.
- **Why it matters**: analytics often contains sensitive product info (usage, paths). Even “aggregates only” can still be sensitive.
- **Suggested fix**: add admin gate similar to other admin functionality; optionally move under `/api/admin/*` and ensure consistent access control.

### MEDIUM-2 — Analytics stats endpoint creates a new Neo4j driver per request (perf + connection churn)

- **Where**: `web-app/app/api/analytics/stats/route.ts:57`
- **What**: `neo4j.driver(...)` is created and closed every request; bypasses the repo’s shared driver pattern (`web-app/lib/neo4j.ts`).
- **Why it matters**: higher latency, connection churn, and risk of exhausting Neo4j Aura connection limits under load.
- **Suggested fix**: reuse `getDriver()/getSession()` from `web-app/lib/neo4j.ts`; ensure `session.close()` is always in `finally`.

### MEDIUM-3 — Analytics parsing is brittle and expensive (string-splitting JSON-like blobs in Cypher)

- **Where**: `web-app/app/api/analytics/stats/route.ts:92`
- **What**: extracts fields from `e.event_data` by splitting strings in Cypher (e.g., `split(data, '"path":"')`).
- **Why it matters**: correctness breaks if encoding changes; string ops in Cypher can be slow; user-controlled strings can poison metrics (even without injection).
- **Suggested fix**: store structured maps in Neo4j (properties), or parse JSON at write time and store normalized fields (`path`, `query`, `results_count`, etc.).

### MEDIUM-4 — Auth UI does not respect “graceful degradation” when OAuth env vars are missing

- **Where**:
  - Providers are conditionally configured: `web-app/lib/auth.ts:57`
  - But buttons always call `signIn('google')` / `signIn('github')`: `web-app/components/AuthButtons.tsx:59`
  - `authConfigured` is exported but unused: `web-app/lib/auth.ts:112`
- **What**: in environments without provider env vars, the UI still shows provider buttons that likely lead to runtime errors / confusing UX.
- **Why it matters**: contradicts “site works identically for anonymous users when auth is unconfigured” and makes staging/prod config mistakes more painful.
- **Suggested fix**: use `authConfigured` (via a server-provided flag or a small `/api/auth/config` endpoint) to hide or disable provider buttons when providers aren’t available.

### MEDIUM-5 — Public collection routes/pages may return Neo4j temporal types to clients (serialization + UI bugs)

- **Where**:
  - Collections list returns `created_at` / `updated_at` from Neo4j: `web-app/app/api/collections/route.ts:20`
  - Profile sorts by `created_at` as a string: `web-app/app/profile/page.tsx:142`
  - Contributions API returns `cb_f.timestamp` / `cb_m.timestamp` directly: `web-app/app/api/user/contributions/route.ts:33`
- **What**: Neo4j temporal values are not guaranteed to be plain strings in JSON. The UI treats them as strings (`localeCompare`), which can break silently.
- **Why it matters**: production runtime errors (“localeCompare is not a function”) and inconsistent ordering.
- **Suggested fix**: normalize temporal values in APIs (e.g., `toString()` or ISO strings) before returning JSON; type them as `string | null`.

### MEDIUM-6 — Collection IDs are predictable (timestamp + short random) for a “public sharing” feature

- **Where**: `web-app/app/api/collections/route.ts:65`
- **What**: IDs embed time and a 5-char random suffix. Combined with public GET, this increases enumeration risk.
- **Why it matters**: makes brute forcing public collections more realistic, especially if IDs ever leak in logs/referrers.
- **Suggested fix**: use UUIDv4 (or crypto-random) for share IDs; consider separate public share tokens.

### MEDIUM-7 — AI endpoints lack input size limits; some log raw model output

- **Where**:
  - Logs raw response: `web-app/app/api/ai/suggest-eras/route.ts:176`
  - Large prompt built from user fields: `web-app/app/api/ai/validate-location/route.ts:68`
- **What**: request fields (`workTitle`, `notes`) aren’t size-limited server-side; raw AI output is logged on parse failures.
- **Why it matters**: cost amplification (very large prompts), log noise/PII retention, and potential prompt injection content in logs.
- **Suggested fix**: enforce max lengths in the API (not only in UI), and avoid logging full model outputs in production (log hashes or first N chars).

## LOW

### LOW-1 — Middleware matches almost all paths but does no work (unnecessary overhead)

- **Where**: `web-app/middleware.ts:10`
- **What**: middleware runs for most requests but only returns `NextResponse.next()`.
- **Why it matters**: small but non-zero overhead; also misleading comment suggests admin gating lives elsewhere.
- **Suggested fix**: remove middleware entirely if not needed, or narrow matcher to actual needs.

### LOW-2 — `CollectionMiniGraph` uses unstable node IDs and recomputes graph data every render

- **Where**: `web-app/components/CollectionMiniGraph.tsx:63`
- **What**: falls back to `Math.random()` for node IDs; graphData computed in an IIFE each render (new object identity each time).
- **Why it matters**: can cause jitter, remounts, and extra force-graph work; makes bugs non-deterministic.
- **Suggested fix**: require stable IDs (or derive from `collection_id + item_id`) and memoize via `useMemo`.

### LOW-3 — Collections “Add” button ignores non-OK responses and provides no feedback

- **Where**: `web-app/components/AddToCollectionButton.tsx:56`
- **What**: on add failure, it silently does nothing (no toast/error), and parses JSON even on non-OK `/api/collections` fetch. `web-app/components/AddToCollectionButton.tsx:46`
- **Why it matters**: confusing UX; harder debugging in production.
- **Suggested fix**: check `r.ok`, show an error state, and handle JSON parsing failures gracefully.

### LOW-4 — `web-app/lib/auth.ts` uses `any` for auth callback inputs

- **Where**: `web-app/lib/auth.ts:5`
- **What**: `upsertUserInNeo4j(user: any, account: any, profile: any)` loses type safety, and makes future refactors risky.
- **Why it matters**: TypeScript quality regression; easier to introduce runtime bugs around `profile` shape.
- **Suggested fix**: use NextAuth types (`User`, `Account`, `Profile`) and narrow per-provider profile fields.
