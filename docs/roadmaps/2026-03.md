# Fictotum — March 2026 Roadmap

## North Star
Fictotum becomes a product people want to share — AI-discovered connections grounded in graph data, automated content growth, and visual richness that makes every figure page worth exploring.

## Milestones

### M1: Fix Deploy Pipeline + Open Pre-Beta — Complete
- **Why it matters**: Can't ship anything if deploys are broken. Opening to a small audience creates accountability.
- **Acceptance criteria**:
  - [x] Resolve FIC-148 (Vercel deploy pipeline not reflecting latest commits) — verified working 2026-02-23, latest commit deployed
  - [x] Verify fictotum.com reflects latest code after a push to main — confirmed commit 21af1a7 deployed, middleware redirects to /password
  - [ ] Share password with 5-10 trusted beta testers — user action, not code
  - [ ] Regenerate Jesus + Peter portraits once Gemini quota resets (FIC-144) — tooling ready, awaiting quota
- **Linear tickets**: FIC-148, FIC-144
- **Key files**: `web-app/vercel.json`, Vercel dashboard project settings

### M2: Discovery Agent — Not Started
- **Why it matters**: The "wow" feature. AI-discovered connections grounded in graph data. This is what makes Fictotum different from a Wikipedia sidebar. Trails competitive analysis (Feb 20) validated the concept — but Fictotum's typed graph is a structural advantage over unstructured text.
- **Acceptance criteria**:
  - [ ] `lib/connection-scoring.ts` — graph-only signals (hop distance, cross-era surprise, sentiment divergence, property completeness)
  - [ ] "Discover Connections" section on figure detail pages (on-demand, not precomputed)
  - [ ] Claude Sonnet 4.6 narration of discovered connections with citations to graph data
  - [ ] Connections feel grounded in historical fact, not arbitrary (the Trails critique)
  - [ ] Test on 5+ figures with known interesting connections
- **Linear tickets**: FIC-120, FIC-118
- **Key files**: `web-app/lib/connection-scoring.ts` (new), `web-app/app/figure/[id]/page.tsx`
- **Dependencies**: FIC-120 (scoring) must be built before FIC-118 (discovery agent)

### M3: Gemini Extraction Pipeline — Not Started
- **Why it matters**: Automates the path from Wikipedia/source text to structured entities in Neo4j. Transforms content growth from manual batch imports to AI-assisted curation. The moat is data — 1,594 nodes is good; 5,000+ with rich interconnections is defensible.
- **Acceptance criteria**:
  - [ ] Stage 1: Gemini extracts structured data (figures, works, portrayals, relationships) from Wikipedia articles
  - [ ] Stage 2: Entity resolution against existing graph (Wikidata Q-ID matching, duplicate detection)
  - [ ] Stage 3: Human review via dry-run output before import
  - [ ] Successfully import one "deep cluster" (e.g., Medieval Europe — FIC-35) through the full pipeline
  - [ ] Output is valid batch-import JSON that passes existing validation
- **Linear tickets**: FIC-113, FIC-35
- **Key files**: `scripts/import/batch_import.py`, `data/` (schemas)

### M4: Illustration System — Not Started
- **Why it matters**: Every figure page has a placeholder. AI-generated portraits make the product visually compelling and shareable. The prompt infrastructure already exists (FIC-144 updated the STYLE_PREAMBLE for complete facial features).
- **Acceptance criteria**:
  - [ ] Gemini image generation integrated into figure detail pages
  - [ ] Style-consistent portraits following the updated STYLE_PREAMBLE
  - [ ] Batch generation for top 50 most-connected figures
  - [ ] Historical accuracy guardrails (period-appropriate clothing, settings)
- **Linear tickets**: FIC-88 (partial)
- **Key files**: `web-app/lib/prompt-templates.ts`, `web-app/app/api/admin/regenerate-image/`

### M5: Orphan Connection + Data Quality — Not Started
- **Why it matters**: 147 figures are dead ends. Connecting them to the graph turns every page into a discovery starting point. No orphans = no bounce pages.
- **Acceptance criteria**:
  - [ ] Triage all 147 orphan figures: connect to existing works, merge duplicates, or remove if unsalvageable
  - [ ] Wikidata link builder suggests connections for orphans automatically
  - [ ] Zero orphan figures remaining in the graph
  - [ ] Total entity count reaches 2,000+
- **Linear tickets**: FIC-108
- **Key files**: `scripts/qa/`, `scripts/import/batch_import.py`

## Dependencies
- M2 (discovery agent) depends on M1 (deploy fix) — need to be able to ship it
- M3 (extraction pipeline) needs Gemini API quota — verify limits before starting
- M4 (illustrations) also uses Gemini image API — may share quota constraints with M3
- M3 and M4 both use Gemini — sequence if quota is constrained, parallelize if not

## Risks
- **Gemini quota limits** could bottleneck both M3 and M4. Mitigation: check quotas before starting, prioritize M3 over M4 if constrained.
- **FIC-148 could be deeper than expected**. Mitigation: timebox to 1 day, escalate to Vercel support if unresolved.
- **Discovery agent quality** — AI connections that feel shallow will hurt credibility (the Trails/HN critique). Mitigation: FIC-120 scoring before FIC-118 generation, ground everything in graph data with citations.
